freeoxide is a one-person initiative with a simple bias: software I depend on should be open — and open software should be built like it's load-bearing.

I build these projects for myself first. The boilerplate I wanted on day one, the CLI I actually needed, the tools I couldn't find. Then I open them, because someone else probably needed them too.

Rust is the floor, not the headline. The point isn't the language — it's the discipline. Code that's tested until it's boring, structured so the next person can read it, and audited pass after pass until there's nothing left to surprise you.

— maintainer, freeoxide

What makes a project qualify

A project doesn't carry the freeoxide name until every box below is true. No exceptions, no "we'll fix it later."

[x]Real test coverage. Not a number to game — paths exercised, edge cases pinned. >90% is the floor, not the goal.
[x]Multi-pass AI audits. Every project gets 4–5 independent audit passes. Each pass is told to find something the last one missed — and usually does.
[x]Architecture review. Module boundaries and structure reviewed the same way. Pass after pass, until the shape stops changing.
[x]Performance passes. Dedicated runs to find hot paths and allocations worth removing. Measured, not guessed.
[x]Idiomatic & documented. Clippy-clean, sensible defaults, a public API you can read. If it needs a comment to be safe, it gets one.
[x]Open by default. MIT / Apache-2.0, readable git history, no surprise dependencies. Auditable all the way down.

How the testing actually works

Two loops running against each other: a human looking for the obvious failures, and AI grinding for the ones a human would never think to write.

manual review

Read every line. Then run it like a user would — and again like an attacker would.

The boring failures get found by hand: bad input, empty state, the unhappy path, the thing that only breaks the third time.

ai-assisted

AI writes tests in rounds — 5 to 8 of them. Each round is asked for the cases the last one missed.

New edge cases, new inputs, new ways to break it. It's slower on purpose. That's the point.

01baseline suitehappy paths + the obvious errors
02round two+ boundary values, empty & max inputs
03round three+ unicode, encoding, locale edge cases
04round four+ concurrency, ordering & race conditions
05rounds five–eight+ the weird ones you'd never write by hand

slower on purpose. no stone left unturned.

Get involved

It's one person right now — which means a single good PR genuinely moves things. Here's where to start.

01Browse what exists.Start in the project list. Read the code, the tests, the audit notes. Open an issue the moment something smells off.
02Raise the bar, don't just add to it.The most welcome PRs add a test, kill an edge case, or make the structure simpler — not just bigger.
03Submit a project.Built something to this standard? Propose it. It runs through the same audit passes before it carries the freeoxide name.
04Build in public with me.Most of this happens in the open. Follow along, argue with the decisions, suggest the next tool worth building.
